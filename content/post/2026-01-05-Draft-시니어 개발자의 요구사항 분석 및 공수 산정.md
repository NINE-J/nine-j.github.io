# 체험단 관리 사이트 외주 한정 질문 시뮬레이션

## 1. 요청자(클라이언트)에게 던지는 핵심 질문

개발자는 구현 가능 여부를 넘어 '비즈니스 예외 상황'을 확인해야 합니다.

- **블로그 품질(지수) 연동 방식:** "블연플 사이트에서 데이터를 자동으로 긁어오길(스크래핑) 원하시나요, 아니면 해당 서비스의 API 키를 제공해주실 수 있나요?"
- **구글폼 항목의 가변성:** "매번 항목이 바뀐다고 하셨는데, DB 구조를 매번 수동으로 바꾸실 건가요 아니면 어떤 항목이 들어와도 유연하게 쌓이는 구조를 원하시나요?"
- **동시 접속 및 보안:** "직원들이 동시에 접속했을 때 데이터 충돌(동시 수정) 방지가 중요한가요?"
- **기존 데이터 이전:** "기존에 노션이나 엑셀에 적어둔 업체 현황 데이터도 새 시스템으로 옮겨야 하나요?"

## 2. 기획서 기준으로 살피는 '개발 포인트'

개발자는 문서의 텍스트 너머의 '데이터 관계'를 먼저 봅니다.

- **다대다(N:M) 관계 설정:** 한 명의 인플루언서가 여러 시술을 중복 지원하고, 여러 지역에 방문 가능하며, 여러 업체(병원)의 캠페인에 참여하는 복잡한 구조를 어떻게 DB 테이블로 쪼갤지 살핍니다.
- **상태 값 정의:** '신규 접수', '확정 후보', '방문 단계', '후기 작성' 등 각 단계의 트리거가 무엇인지 확인합니다.
- **UI/UX 구조:** 사이드바 메뉴 구성과 클릭 시 나타나는 대시보드의 데이터 필터링 조건(지역별, 업체별 등)이 기술적으로 성능 저하 없이 구현 가능한지 검토합니다.

## 3. 개발 난이도, 공수 및 비용 산정 로직

전문 개발자는 '페이지 수'가 아니라 '기능의 깊이'로 견적을 냅니다.

- **난이도 (중상):** 단순 게시판보다는 높습니다. 구글 API 연동, 외부 사이트 데이터 연동(블연플), 직원별 권한 로직 때문입니다.
- **공수 산정:**
    - **DB/API 설계 (30%):** 구글폼 데이터 수급 및 중복 필터링 로직 구축.
    - **백엔드/관리 로직 (40%):** 업체별/지역별/직원별 데이터 필터링 및 권한 처리.
    - **프론트엔드 (30%):** 대시보드 UI 및 상세 팝업창 구현.
- **비용 산정:** 직접 코딩(Custom Build) 시 최소 1,500만 원 이상에서 시작하며, 앞서 언급한 노코드(Low-code) 툴 활용 시 800~1,000만 원 선에서 제안할 가능성이 큽니다.

## 4. 개발자의 마인드셋 (PM vs 사업가 vs PoC)

능력 있는 풀스택 개발자는 단순히 '코드 짜는 사람'이 아닌 **솔루션 제공자**로 임합니다.

- **사업가 마인드:** "직원이 수동으로 정리하는 시간을 줄여서 인건비를 얼마나 아낄 수 있는가?"에 집중합니다. 14 따라서 화려한 디자인보다 **데이터 입력 속도와 정확도**를 우선시합니다.
- **PM/PoC 우선주의:** 처음부터 모든 기능을 다 만들기보다, "**구글폼 → DB 자동 전송 → 블로그 지수 자동 조회**"라는 핵심 루프가 제대로 도는지 확인하는 **PoC(개념 증명) 단계를 먼저 제안**할 것입니다.
- **확장성 고려:** 업체가 계속 늘어날 수 있다는 점을 고려하여, 관리자가 코딩 없이 새 업체를 등록하면 자동으로 필터 메뉴가 생성되는 시스템을 고민합니다.

## 결론: 이런 개발자가 필요합니다

이 프로젝트는 "데이터 정규화(Normalization)"를 이해하고 있으며, 구글 워크스페이스 생태계와 외부 웹 데이터를 연동해 본 경험이 있는 개발자가 적합합니다.
